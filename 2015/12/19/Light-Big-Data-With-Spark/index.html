Light Big Data With Apache Spark

2015-12-19

Ten tools for ten big data areas 03_Apache Spark from Will Du

Above presentation is the third topic I have covered for series of talks about the Ten Tools for Ten Big Data Areas. Apache Spark is one of the greatest big data open source projects in nowadays.

There are so many articles, reports about how great the Apache Spark it is. I also heard lots of people keep saying that Spark will replace Hadoop and do everything in big data. As for me, I agree on some of them by saying Spark does a great job on some areas. However, I do disagree some saying about spark and I’ll talk about this today as most people do not realise there Spark may not work very well in some areas.

First, Spark is not going to replace Hadoop. Hadoop is a big data platform while Spark is an application. For another saying in terms of IPhone ecosystem, it is a relationship between IOS and Apps. Even Spark is powerful enough to do lots of things in big data, it cannot “rules all”. We still have used cases which require other applications. Hadoop, especially Yarn, gradually become more and more important in the role of a platform by providing a multi-purpose universal platform for run various of big data applications.

Second, Spark is not good for everything. Below are some areas that I think we have other better options than using Apache Spark.

  • Spark has to rely on HDFS or other to store data. It is a mainly computing engine. Spark is based on RDD, which is the immuable data set. As a result, Spark does not fit for the use case when you need to modify the data.
  • Spark uses micro-batch execution model to offer streaming function. As a result, it has a limitation when the stream interval as low as 0.5 seconds. Therefore, it is not a good option for real-time streaming scenario.
  • Spark runs on the JVM and leverages Java’s garbage collector. As JVM is designed for its general purpose, it lacks flexiablilty, good user experience, as well as memory usage overhead. Spark team has realised about this and comes up the Project Tungsten, which creates Spark its own memory management system.
  • For data ETL (extract, transformation, and load), you may not need Spark’s speed, but more focus on the reliability as well as failure recovery. In this case, MapReduce’s processing style can be just fine as stable batch-mode processing.
  • Spark aims to replace MapReduce, but it does not provide any way to back compatible with old MapReduce jobs. However, there are legacy MapReduce jobs we cannot get rid of immediately.
    Machine learning library still needs improvement by supporting more algorithm as well as accuracy GrapX is still not mature (comparing (Apache Giraph)[http://giraph.apache.org/]) and some functions are only available in Scala API.

Although some improvement areas are listed above, there is no doubt Apache Spark is greatest and hot big data application stack for now. I do believe it has good future.
